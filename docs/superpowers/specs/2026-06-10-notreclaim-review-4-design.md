# NotReclaim Review 4 — design (within-column priority drag + planner polish)

**Date:** 2026-06-10. **Source:** user feedback after Review 3. Branch `feat/review4-ordering`.

| # | Item | Decision |
|---|------|----------|
| 1 | Click-to-create task/event windows too small | `CreatePopover` grows to a fixed `w-[340px]` card (it was `left-1 right-1` inside a ~130px day column), padding/type scale up, and it anchors left for Mon–Thu columns / right for Fri–Sun so it never overflows the grid edge (`align` prop from WeekGrid's `dayIndex`). |
| 2 | Planner blocks show `<Task name>: <Subtask name>` | Display-only relabeling in the Planner: for each task, its **not-done subtasks in creation order** are assigned to that task's blocks **in start-time order**; block *i* shows `Task: Subtask_i`. Blocks beyond the open-subtask count (or tasks without open subtasks) keep the plain task title. Pure helper `app/planner/blockLabels.ts`, applied in `Planner.tsx` before passing blocks to WeekGrid (WeekGrid untouched; drag/commit use block ids, so labels are cosmetic). Done subtasks are skipped — finished work shouldn't claim future blocks. |
| 3 | Drag tasks WITHIN a column to set priority (Review-1 F2) | New **`Task.sortOrder Float @default(0)`**. Ordering within a bucket = `sortOrder` asc, then `dueBy` asc (existing rows keep 0 → legacy dueBy order until first drag). Engine: `FlexibleTask.sortOrder?` and the work sort becomes `priority → order → tie → id` where `order` = task `sortOrder ?? 0`, habits 0 (scheduler tests don't constrain same-priority task/habit interleave). Board: row-level drag-over tracks an insertion index (top/bottom half of each row, end-of-list zone); drop computes the new `sortOrder` as the **midpoint of the visible neighbors** (top: first−1, bottom: last+1, empty column: 0) and PATCHes `{sortOrder, priority?}` — same-bucket reordering is now allowed (was a no-op). Insertion indicator: indigo line at the insertion point. Neighbor lookup uses the VISIBLE (filtered) column list — what the user sees is what they insert between. |
| 4 | Remove the priority number from task configuration | The drawer's "Priority (lower = scheduled first)" number input is removed; `taskForm` drops `priority` from state/patch. The bucket is controlled only by board drag (column = bucket, position in column = order). |
| 5 | New tasks default to the lowest priority within their column | `TaskRepository.create`: when `sortOrder` is not provided, it defaults to `max(user's sortOrder) + 1` — below every existing task in any column. Applies to the New Task modal and the click-to-create popover (neither sends sortOrder). |

**Out of scope:** persisting an order for habits; cross-user concerns; renumbering/compaction of midpoint floats (doubles give ~50 levels of halving between neighbors — fine at this scale); optimistic reorder (refetch round-trip is acceptable, consistent with onComplete).
